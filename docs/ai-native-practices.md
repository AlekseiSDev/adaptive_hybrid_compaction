# AI-Native Repository — практики и приоритизация

> Документ описывает, как превратить обычный репозиторий в AI-native среду — такую, где кодинг-агенты работают надёжно, точечно и почти без ручных правок. Практики разбиты по уровням и ранжированы по приоритету внедрения; экспериментальные явно помечены.

---

## Введение: AI-native development & harness engineering

**AI-native development** — сдвиг практик команды и репозитория, при котором кодинг-агенты становятся первоклассными контрибьюторами наравне с людьми. **AI-native repo** — репозиторий, организованный так, чтобы агент в нём работал максимально продуктивно: предсказуемая структура, явные интерфейсы, проверяемые механически инварианты, документированный контекст в самих артефактах.

**Harness engineering** — инженерная дисциплина внутри этого сдвига: проектирование runtime-окружения агента (промпты, инструменты, hooks, sandbox, sub-agents, skills, контекстные политики, evals). Прижившаяся в комьюнити формула — **Agent = Model + Harness**: всё, что не модель, но имеет отношение к runtime агента, относится к харнесу.

Соотношение примерно такое:

- **AI-native repo** отвечает на вопрос: _как устроен мир, в котором работает агент?_ — структура файлов, документация, тесты, governance.
- **Harness engineering** отвечает на вопрос: _как агент действует внутри этого мира?_ — какие инструменты у него под рукой, как сжимается контекст, как сабагенты делятся работой, какие проверки автоматически запускаются после изменений.

Эти две темы пересекаются, но не идентичны. AI-native — организационный и архитектурный сдвиг. Harness engineering — техническая дисциплина, плотная часть этого сдвига.

Ключевой переворот приоритетов, который зафиксировали [GitHub Copilot team](https://github.blog/ai-and-ml/github-copilot/agent-driven-development-in-copilot-applied-science/) и [OpenAI](https://openai.com/index/harness-engineering/) на эксперименте с миллионом строк Codex-кода: рефакторинги, документация, тесты, чистка dead code — всё, что традиционно жило в "когда-нибудь потом", в AI-native становится **критическим путём**. Без этого агент работает плохо и накапливает долг; с этим — масштабируется почти линейно по человеко-часам.

### Что в этом документе

Документ устроен так:

- Сначала рассматриваем обе темы — **AI-native repository** (организация кода под агента) и **engineering harness** (чеклист практик от T0 до T4).
- Потом разбираем, что конкретно из этого **забираем в JayFlow v3** — какие практики берём в работу сейчас, какие позже, какие не трогаем.
- Дальше — реальные проблемы команды (Concerns) и как мы их лечим через перечисленные практики.
- В конце — операционка: командные и индивидуальные overrides, метрики, антипаттерны.

### Главный тезис

Идея не в том, чтобы написать хороший `CLAUDE.md` и завести skills для ключевых интеграций — это необходимо, но недостаточно. AI-native development — **это подход к организации кода и репозитория** (структура, документация, mechanical rules, явные интерфейсы), без которого харнес работает на 30% мощности. А engineering harness — **не статический набор практик, а эволюционирующая система**: новые skills для появляющихся доменов, индексирование растущей документации, отлов повторяющихся ошибок через правила или линтеры, удаление scaffolding'а, который стал dead code после апгрейда модели. Эта эволюция — постоянная работа в фоне, а не разовый проект.

## AI-native repository: организация кода

Agent legibility частично совпадает с human readability, но с другими акцентами. OpenAI в [Harness engineering](https://openai.com/index/harness-engineering/) формулируют это как: репозиторий — единственный контекст для агента; всё, что важно для работы, должно жить в версионируемых артефактах, а не в чатах или головах людей. GitHub Copilot team в [Agent-driven development](https://github.blog/ai-and-ml/github-copilot/agent-driven-development-in-copilot-applied-science/) добавляют: главная ценность для команды — приоритизация рефакторингов, тестов и документации, потому что именно они делают агент эффективным.

| Аспект | Human readability | Agent legibility |
|---|---|---|
| Что важно | Элегантность, минимализм | Предсказуемость, явность |
| Файлы | Размер второстепенно | Маленькие файлы лучше — меньше токенов на чтение |
| DRY | Часто хорошо | Иногда вредит — дублирование лучше неявной зависимости |
| Конвенции | Можно гибко | Жёсткие, проверяемые механически |
| Документация | Чем меньше, тем лучше | Чем больше структурированной — тем лучше |

### Шесть свойств agent-friendly модуля

1. **Высокая когезия** — модуль решает одну ответственность. Агент при изменении не должен ходить по всему репо.
2. **Низкое связывание через явные интерфейсы** — `index.ts`, `__init__.py`, `mod.rs` явно перечисляют публичный API. Внутренности модуля скрыты.
3. **Однонаправленные зависимости** — зависимость идёт по слоям (например, `domain → application → infrastructure`, а не наоборот). Проверяется механически, не промптом.
4. **Co-located unit-тесты и документация** — `file.test.ts` рядом с `file.ts`, `README.md` в каждом нетривиальном модуле. Cross-module integration и e2e тесты — отдельно в общей `tests/`-иерархии, потому что они не принадлежат конкретному модулю.
5. **Self-describing структура** — имена файлов и директорий говорят, что внутри. Агент находит по соглашению, а не по grep-у всего проекта.
6. **Маленький размер файла** — практический ориентир: до 300–500 строк. Большие файлы заставляют агента грузить много токенов на маленькое изменение и ломают точечные правки.

### Эталонная структура (Fastify / TypeScript пример)

```
project/
├── AGENTS.md                    # глобальные правила
├── README.md
├── package.json
├── tsconfig.json
├── eslint.config.js             # source of truth для стиля
├── docs/
│   ├── index.md                 # документ-индекс репо
│   ├── architecture.md          # архитектурные инварианты
│   ├── decisions/               # ADR
│   ├── features/                # spec-документы по сложным фичам
│   └── runbooks/                # operational
├── src/
│   ├── domain/                  # чистые сущности, без I/O
│   │   ├── index.ts             # явный public API (re-exports)
│   │   ├── README.md            # что здесь и почему
│   │   ├── user.ts
│   │   ├── user.test.ts         # co-located unit-тест
│   │   └── ...
│   ├── application/             # use cases, оркестрация
│   │   ├── AGENTS.md            # local override для слоя
│   │   ├── index.ts
│   │   ├── createUser.ts
│   │   ├── createUser.test.ts
│   │   └── ...
│   ├── infrastructure/          # I/O, БД, внешние API
│   │   ├── index.ts
│   │   ├── userRepository.ts
│   │   ├── userRepository.test.ts
│   │   └── ...
│   └── interfaces/              # Fastify plugins, routes
│       ├── plugins/
│       └── routes/
├── tests/                       # cross-module: integration и e2e
│   ├── integration/
│   └── e2e/
└── scripts/
    └── verify.sh                # одна команда — все checks
```

Ключевые свойства этой структуры:
- Зависимости направлены внутрь: `interfaces → application → domain`. Проверяется механически (см. T1.3).
- В каждом слое — `index.ts` с явными re-exports; внутренности модуля приватны по соглашению.
- В каждом нетривиальном модуле — локальный `README.md` с контекстом для агента.
- Unit-тесты co-located с тестируемым кодом; integration и e2e — в `tests/`.
- Где правила слоя отличаются от глобальных — локальный `AGENTS.md` (Codex и Claude Code оба понимают иерархию overrides).
- `scripts/verify.sh` — единая точка входа для всех checks; агенту не нужно гадать, что запускать.

## Engineering harness: чеклист практик

Ранжирование от **T0 (без этого AI-native не работает)** до **T4 (экспериментальное, мониторим)**.

### Tier 0 — Foundation (без этого нет смысла начинать)

#### T0.1 — `AGENTS.md` или `CLAUDE.md` в корне
Минимальный набор:
- Структура проекта (какие директории за что отвечают, где живёт что).
- Build & test команды (`./scripts/verify.sh`) — чтобы агент знал, как проверить свои изменения.
- Pointer на конфиги линтера/форматтера/typecheck — они source of truth для стиля и naming, дублировать не нужно.
- Project-specific конвенции, которые линтер не ловит: например, "новые routes регистрируются через X plugin", шаблон именования use cases.
- Anti-patterns: что точно нельзя (накапливается со временем, см. T1.4).
- Указатели на ключевые файлы (`docs/index.md`, `docs/architecture.md`, `docs/decisions/`).

**Эффект**: агент перестаёт каждый раз заново выяснять, как тут принято.
**Источник**: [The Agent-Native Repo](https://www.harness.io/blog/the-agent-native-repo-why-agents-md-is-the-new-standard); [OpenAI AGENTS.md guide](https://developers.openai.com/codex/guides/agents-md).

#### T0.2 — Запускаемые `verify`-команды одним вызовом
Агент должен иметь возможность одной командой проверить, что его изменения не сломали мир: lint + typecheck + unit tests + быстрый smoke. Если для проверки нужно вручную открывать пять консолей — агент не будет проверять, и регрессии накопятся.

**Эффект**: tight feedback loop без участия человека.
**Источник**: [The Agent-Native Repo](https://www.harness.io/blog/the-agent-native-repo-why-agents-md-is-the-new-standard) (паттерн `code-change-verification`).

#### T0.3 — Hooks для автоматических проверок после изменений
Каждый раз, когда агент меняет файл — автоматически прогоняется минимальный набор checks, и результат возвращается в контекст. В Claude Code — через `hooks` в `.claude/settings.json`.

**Эффект**: ошибки ловятся в момент создания, а не через 10 шагов.
**Источник**: [Anthropic — Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents).

#### T0.4 — Sandbox / permissions
Что агент может делать без подтверждения, что — с подтверждением, что — никогда. Особенно для destructive операций (rm -rf, force push, прод-БД).

**Эффект**: страховка от катастроф.
**Источник**: [Avi Chawla — The Anatomy of an Agent Harness](https://blog.dailydoseofds.com/p/the-anatomy-of-an-agent-harness).

#### T0.5 — Предсказуемая структура директорий
Если агент не может вывести из структуры, куда положить новый код — он положит его где попало. Convention over configuration. Соглашения должны быть либо общеизвестными (стандарт фреймворка), либо явно описаны в AGENTS.md.

**Эффект**: новый код попадает в правильное место без подсказок.
**Источник**: [GitHub Blog — Agent-driven development](https://github.blog/ai-and-ml/github-copilot/agent-driven-development-in-copilot-applied-science/).

### Tier 1 — Высокий приоритет (внедрять в первые недели)

#### T1.1 — Co-located документация модулей
В каждом нетривиальном модуле — короткий `README.md`: зачем модуль, что он делает, что не делает, кто его использует, нюансы. Не для красоты документации, а как контекст для агента, когда он туда заходит.

**Эффект**: агент понимает контекст модуля без чтения всего кода вокруг.
**Источник**: [GitHub Blog — Agent-driven development](https://github.blog/ai-and-ml/github-copilot/agent-driven-development-in-copilot-applied-science/).

#### T1.2 — Architecture Decision Records (ADR / `docs/decisions/`)
Короткие markdown-записи о ключевых решениях: что выбрали, почему, какие альтернативы рассматривали. Агент читает их, когда нужно понять "почему так, а не иначе".

**Эффект**: агент перестаёт переоткрывать обсуждённые трейдоффы.
**Источник**: индустриальная практика ADR (Michael Nygard, 2011); в AI-native контексте — [OpenAI Harness engineering](https://openai.com/index/harness-engineering/) (executable plans как версионируемые артефакты).

#### T1.3 — Mechanical rules для архитектурных инвариантов
Архитектурные инварианты, которые **нельзя нарушить**, не должны жить в промпте. Они должны проверяться:
- Custom линтерами (например, проверка что `domain/` не импортирует из `infrastructure/`).
- Structural tests (тесты, проверяющие свойства графа зависимостей).
- Pre-commit hooks или CI gates.

В JS/TS — `dependency-cruiser`, `eslint-plugin-boundaries`. В Python — `import-linter`, `pytest-archon`.

**Эффект**: агент физически не может закоммитить нарушение.
**Источник**: [OpenAI Harness engineering](https://openai.com/index/harness-engineering/) (mechanical rules через детерминированные линтеры); [Martin Fowler — Harness engineering](https://martinfowler.com/articles/harness-engineering.html).

#### T1.4 — `AGENTS.md` как living anti-pattern log
Принцип Хашимото: каждый раз, когда агент совершает ту же ошибку второй раз — добавляй правило. AGENTS.md превращается из документации в корпоративную память факапов.

**Эффект**: со временем повторные ошибки исчезают.
**Источник**: [Mitchell Hashimoto — My AI Adoption Journey](https://mitchellh.com/writing/my-ai-adoption-journey); публичный пример — AGENTS.md в [Ghostty terminal](https://github.com/ghostty-org/ghostty).

#### T1.5 — Явные `index.ts` / `__init__.py` с public API
Внутри модуля видно всё; снаружи — только то, что в `index.ts` exports или `__all__`. Агент при использовании модуля видит только то, что должен.

**Эффект**: low coupling enforced механически.
**Источник**: индустриальная практика (Parnas, "On the Criteria To Be Used in Decomposing Systems into Modules", 1972); в AI-native контексте — минимизация surface area модуля сокращает токены контекста.

#### T1.6 — Документ-индекс репозитория (`docs/index.md`)
Один файл-карта: где какие документы, какие модули за что отвечают, ссылки на ADR, runbooks, спеки фич. CLAUDE.md / AGENTS.md ссылаются на этот index первой строкой; агент при старте сессии получает карту репо в ~200 строк вместо grep-сканирования.

Это не embedding-index (см. T4 для семантического поиска) — это organize-the-repo подход: репо структурирован для агента.

**Эффект**: агент не теряется в большом репо, навигация дешёвая.
**Источник**: [OpenAI Harness engineering](https://openai.com/index/harness-engineering/) (репо как единственный контекст агента, organized for agent legibility).

#### T1.7 — Spec-first и tracking для сложных или долгих фич
Для сложных или multi-session фич — отдельный spec-документ плюс progress log. Часто оба живут в одном файле `docs/features/<feature>.md`:

- **Spec** (input для работы): что строим, критерии приёмки, что out-of-scope. Агент читает спеку первой, не выводит реквайерменты из своей головы.
- **Progress** (state работы): что сделано, что осталось, какие препятствия. Свежая сессия быстро понимает, где работа остановилась.

Для большинства задач достаточно встроенного `/plan`. К отдельному spec-документу прибегаем для сложных или multi-session фич, где критерии приёмки нужно зафиксировать как версионируемый артефакт.

**Эффект**: критерии "сделано" фиксированы заранее, агент не разъезжается, ревью валидно.
**Источник**: [Anthropic — Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) (initializer agent + claude-progress.txt); [Anthropic — Harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps) (planner с structured spec).

### Tier 2 — Средний приоритет (для зрелых проектов)

#### T2.1 — Иерархический `AGENTS.md` с overrides
В корне — общие правила, в `src/payments/` — специфичные для платежей правила, переопределяющие общие. Codex и Claude Code понимают эту иерархию.

**Эффект**: правила локализованы там, где применяются.
**Источник**: [OpenAI AGENTS.md — cascading rules](https://developers.openai.com/codex/guides/agents-md).

#### T2.2 — Skills с progressive disclosure
Вместо того чтобы пихать все инструкции в system prompt, выносим вертикальные знания в Claude Skills (или эквивалент в Codex/OpenCode). Агент подгружает skill только когда задача релевантна — instruction budget не сгорает на старте.

Полезные категории skills для команды:

- **Module-specific skills** — для каждого нетривиального модуля свой skill: публичный API, паттерны обработки ошибок, конвенции, ссылки на reference-файлы. Подгружается, когда агент работает в директории модуля.
- **Vendor/integration skills** — для интеграций с внешними системами: `langfuse-data-extraction.skill.md`, `litellm-config.skill.md`, `alembic-migrations.skill.md`. Накапливают рабочие паттерны, которые сложно вывести из публичной документации провайдера.
- **Workflow skills** — для типовых процессов: `pr-review.skill.md`, `new-feature-scaffolding.skill.md`. Описывают последовательность шагов.
- **Style/taste skills** — командные или личные предпочтения, которые трудно зафиксировать в линтере: "когда писать комментарии", "когда абстракция уместна". Командные живут в репо, личные — в `~/.claude/skills/`.

**Эффект**: instruction budget не сгорает на старте; модуль- и vendor-специфичные знания доступны ровно тогда, когда они нужны.
**Источник**: [Anthropic — Introducing Agent Skills](https://www.anthropic.com/news/skills); [Addy Osmani — Agent Harness Engineering](https://addyosmani.com/blog/agent-harness-engineering/) (~95% экономии контекста через lazy loading в Claude Code); [HumanLayer — Skill Issue](https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents) (включая предупреждение о malicious skills).

⚠ **Безопасность**: skill-регистры (ClawHub, skills.sh) уже ловили на распространении malicious skills. Внешние skills — как `npm install random-package`, читать что ставишь.

#### T2.3 — Sub-agents как context firewall
Делегировать поисковые / исследовательские / разовые задачи в изолированные sub-agent-ы. Parent держит план, дочерние возвращают только структурированный результат.

**Эффект**: parent context не засоряется промежуточным шумом.
**Источник**: [HumanLayer — Skill Issue](https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents) (sub-agents как context firewall).

#### T2.4 — Single point of truth для domain-знаний в репо
Принцип OpenAI: если знание не в репо — для агента оно не существует. Постепенно мигрировать всё важное из чатов и Google Docs в версионируемые артефакты.

**Эффект**: агент имеет тот же контекст, что и senior dev.
**Источник**: [OpenAI Harness engineering](https://openai.com/index/harness-engineering/) (illegibility принцип).

#### T2.5 — Two-agent или three-agent harness для long-running задач
Для задач, занимающих >2 часов: separate planner / coder / evaluator. Применимо для генерации фич end-to-end, миграций, крупных рефакторингов.

**Эффект**: качество и связность на длинной дистанции.
**Источник**: [Anthropic — Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents); [Anthropic — Harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps).

#### T2.6 — Понимание context management в выбранном агенте
Поскольку основная ставка — Claude Code (или Codex там, где удобнее), реализовывать tool-call offloading и compaction руками не нужно — это built-in. Команде стоит знать, как это работает: когда срабатывает автокомпакция, как это конфигурируется через `.claude/settings.json`, когда лучше запустить новую сессию вручную.

**Эффект**: длинные сессии не падают на лимитах, команда понимает поведение инструмента.
**Источник**: [Addy Osmani — Agent Harness Engineering](https://addyosmani.com/blog/agent-harness-engineering/) (compaction и tool-call offloading как примитивы).

---

### Tier 3 — Продвинутые практики (для команд, серьёзно инвестирующих в AI-native)

#### T3.1 — Generator-Evaluator паттерн
Один агент генерирует код / дизайн / текст, второй судит по чётким критериям и возвращает структурированную критику. Anthropic применяют до 15 итераций. Полезно для задач со субъективной оценкой и для верифицируемых, но сложных задач.

**Эффект**: качество на уровне 1–2 итераций ручного review.
**Источник**: [Anthropic — Harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps).

#### T3.2 — Parallel agents для разнотипных задач
Запускать в параллель несколько агентов: один на triage issues, другой на code review, третий — на исследование. Координация через GitHub / файлы / events.

**Эффект**: лучшее использование человеческого времени.
**Источник**: [Mitchell Hashimoto — My AI Adoption Journey](https://mitchellh.com/writing/my-ai-adoption-journey).

#### T3.3 — Garbage collection агенты на расписании
Фоновые таски на расписании сканируют репо на отклонения от принципов и автоматически открывают refactoring PR: чистка dead code, мелкие рефакторинги, актуализация документации. Лучше работает как мелкие непрерывные платежи, чем как накапливающийся долг.

**Эффект**: энтропия не накапливается, рутинная гигиена не зависит от человеческого внимания.
**Риски**: спам PR, конфликты с активной разработкой — нужна зрелая CI и отдельная ветка.
**Источник**: [OpenAI Harness engineering](https://openai.com/index/harness-engineering/) (большинство таких PR мерджатся автоматически за минуту).

#### T3.4 — Documented agent personas / roles
В `AGENTS.md` или отдельных файлах — описание разных режимов работы агента: `code-review-agent.md`, `migration-agent.md`. Каждый — со своими правилами, инструментами, evaluation criteria.

**Эффект**: специализация без размывания базовых правил.
**Источник**: комьюнити-практика (`.claude/agents/` в Claude Code).

### Tier 4 — Экспериментальные практики ⚠

> Эти практики обсуждаются в комьюнити и применяются на отдельных проектах, но не имеют достаточно публичной валидации. Применять с осторожностью, мониторить эффект.

#### T4.1 ⚠ — Telemetry & observability для агентских действий
Логирование: какие tool calls делал агент, сколько токенов, на чём падал, сколько итераций до успеха. Источник данных для улучшения харнеса.

**Риски**: ROI растёт только с масштабом команды; неочевидно что логировать.
**Источник**: [Avi Chawla — Anatomy of an Agent Harness](https://blog.dailydoseofds.com/p/the-anatomy-of-an-agent-harness).

#### T4.2 ⚠ — Evals на critical paths
Не SWE-bench целиком, а свой набор: 20–50 типичных задач твоего проекта, на которых тестируешь любые изменения харнеса.

**Риски**: дорого поддерживать в актуальном состоянии; для маленьких команд evaluation criteria часто меняются быстрее.
**Источник**: [OpenAI Harness engineering](https://openai.com/index/harness-engineering/).

#### T4.3 ⚠ — Three-agent fully-autonomous pipeline (Anthropic style)
Planner + Generator + Evaluator работают часами без человека. Anthropic показали это на full-stack web apps, но публичных воспроизведений на медиум-проектах мало.

**Риски**: высокая стоимость, трудно отладить ошибки в эвалуации.
**Источник**: [Anthropic — Harness design for long-running apps](https://www.anthropic.com/engineering/harness-design-long-running-apps).

#### T4.4 ⚠ — Ralph Loop / continuous iteration hooks
Hook перехватывает попытку агента завершиться и реинъектит prompt в свежем окне до достижения completion goal.

**Риски**: бесконечные циклы, прожигание бюджета.
**Источник**: [Geoff Huntley — The Ralph Wiggum Method](https://ghuntley.com/ralph/).

#### T4.5 ⚠ — Semantic codebase index как tool
Поверх built-in поиска (Grep/Glob) — отдельный embedding-индекс репо, доступный агенту как MCP tool: `find_similar(description, top_k)`. Покрывает кейс, который Grep не ловит — функции с разным неймингом, но одной семантикой.

**Риски**: лишняя инфраструктура; embeddings становятся stale; для небольших репо T1.6 (документ-индекс) даёт 80% той же ценности без сложности.
**Источник**: точечные комьюнити-имплементации, без устоявшейся практики.

#### T4.6 ⚠ — AI-generated AGENTS.md из анализа кодбазы
Codex / Claude Code сами генерируют первичный AGENTS.md, читая код и выводя конвенции. На зрелых проектах — нормально; на хаотичных — закрепляет существующий хаос.

**Риски**: автогенерация может закодифицировать anti-patterns.
**Источник**: [OpenAI Harness engineering](https://openai.com/index/harness-engineering/).

#### T4.7 ⚠ — Agent-to-agent (A2A) protocols, multi-agent ecosystems
Несколько специализированных агентов общаются через стандартизированный протокол. Production-применения пока единичны.

**Риски**: сложность отладки, накопление ошибок, нелинейная стоимость.
**Источник**: [Google A2A Protocol](https://google.github.io/A2A/).

#### T4.8 ⚠ — Auto-deploy custom skills / dynamic skill registry
Skills пишутся самими агентами и автоматически добавляются в registry. Циклически: агент совершил ошибку → скрипт сгенерировал skill → skill доступен следующим сессиям.

**Риски**: качество автогенерируемых skills, прорыв при ошибках в самой генерации skills.
**Источник**: комьюнити-эксперименты (skills.sh, ClawHub).

## Что мы возьмём в JayFlow

Берём в работу обе темы — организацию репозитория и harness-практики уровней T0–T3. Не как разовый проект "перевести репо за неделю", а как инкрементальную дисциплину, которая встраивается в обычный рабочий процесс.

### Часть 1: Практики организации репозитория (AI-native style)

Это всё, что описано выше в разделе **AI-native repository: организация кода**:

- **Принципы agent legibility** (отличия от human readability) — все перечисленные в разделе, по согласованию с архитектором и RnD LLM lead.
- **Шесть свойств agent-friendly модуля** — все перечисленные в разделе (когезия, явные интерфейсы, однонаправленные зависимости, co-located unit-тесты, self-describing структура, маленькие файлы), по согласованию с архитектором и RnD LLM lead.
- **Эталонная структура** для нового кода — Fastify/TypeScript layered (`domain → application → infrastructure → interfaces`) или эквивалент для других стеков. Применяется к новым проектам и при значительных рефакторингах существующих.

Это не разовая задача "перенести структуру за спринт". Это **критерий приёмки для нового кода и фактор приоритизации рефакторингов**: PR не проходит ревью, если новый модуль явно нарушает эти принципы; рефакторинги с большим вкладом в agent legibility поднимаются в приоритете.

### Часть 2: Engineering harness — что берём из чеклиста

**Tier 0 — фундамент.** Это уровень "без него всё остальное не работает". `AGENTS.md` / `CLAUDE.md`, `scripts/verify.sh`, hooks, sandbox, предсказуемая структура — это то, что окупается с первого дня. Без T0 любые продвинутые практики работают на 30% мощности, потому что агенту нечем верифицировать собственную работу.

**Tier 1 — зрелость репо.** Co-located документация, ADR, mechanical rules для архитектурных границ, anti-pattern log, явные public APIs модулей, документ-индекс, spec-first и tracking. Это то, что превращает кодовую базу из "место, где живёт код" в **рабочее окружение для агента и человека одновременно**. Особо важны для нас:

- **T1.3 (mechanical rules)** — `dependency-cruiser` с правилами слоёв снимает с человеческого ревью большую часть проверок "не лезет ли код туда, куда не должен" (см. Concerns ниже).
- **T1.4 (anti-pattern log)** — единственный способ накапливать корпоративную память факапов так, чтобы новый разработчик / новая сессия агента не повторяли уже пройденные грабли. У нас log живёт в отдельном файле `docs/agent-pitfalls.md`, а не внутри `AGENTS.md`/`CLAUDE.md` — детали и обоснование в `docs/fdr/008.fdr.ai-native-adoption.md`.
- **T1.6 (документ-индекс)** — у нас уже есть много документов; нужно собрать их в один index с явными ссылками, на который агент опирается со старта сессии.
- **T1.7 (spec-first и tracking)** — критически важно для валидности ревью на сложных фичах (см. Concerns).

**Tier 2 — работа с агентом как с сотрудником.** Иерархический AGENTS.md, skills, sub-agents, перенос знаний из чатов в репо, two/three-agent harness, понимание context management. Это то, что начинает давать выгоду, когда репо уже зрелое и команда устаканилась с базовыми практиками. Принято в `docs/fdr/009.fdr.ai-native-t2.md`: пилот per-module AGENTS.md (`storage`, `web-api`, `node-ops`), пять skills для core-доменов (БД, graph-ops, codegen, тесты, PR self-review), sub-agents policy в root, sync-check root-пары через `lint-staged`. T2.6 (context management) — без артефакта, дублировал бы vendor-документацию. Custom multi-agent pipeline отложен до конкретной боли.

**Tier 3 — продвинутые практики.** Generator-Evaluator, parallel agents, garbage collection, agent personas. Внедряем выборочно, по мере того как видим конкретный value на конкретных проблемах. Не самоцель.

**Tier 4 — пока не трогаем.** Telemetry, evals, fully-autonomous pipelines, Ralph Loops, A2A protocols, auto-skills, semantic codebase index. Возможно, к чему-то из этого вернёмся через 3–6 месяцев, когда нагрузка вырастет и появится понятный ROI на наблюдаемость и регрессионные evals. Пока — нет.

Ритм внедрения: T0, T1 — сразу, T2 — встраиваем большинство пунктов сразу, T3 - поэтапно и частично по мене необходимости.

**Дополнительно**

Темы, которые обсудили, но пока не оформили как готовые практики уровня T0–T3 — открытые направления для последующей проработки.

#### Документация: soft-link с кодом, без хардового дублирования

Базовая позиция: docs описывают **зачем** и **как** устроено, код описывает **что** делает. Документы не должны дублировать сигнатуры, типы, реализацию — это пересечение с кодом, которое неизбежно отстаёт и превращает доку в ложный источник правды.

Что обдумываем:

- Где проходит граница между co-located README модуля (T1.1) и кодом — в README назначение, контракт с потребителем, неочевидные инварианты; в коде implementation. Дубль API в README — сразу же deprecated.
- Автогенерация части доков (публичный API из `index.ts`, схемы из типов, список команд) — чтобы документ физически не мог отстать. Soft-link через генератор, а не hard-copy руками.
- Skill `module-docs-review` для написания и ревью доков по модулю: единый формат, проверка соответствия текущему коду, обнаружение устаревших разделов. Кандидат на T2.2 (workflow skill).

#### Fast-review внутри команды

Идея: внутрикомандный review должен быть быстрым и преимущественно автоматизированным, чтобы не упираться в человеческое внимание (см. Случай Б3 про невалидное ревью).

Слои, которые рассматриваем:

- **Линтеры и тесты как gate** — всё, что ловится автоматически, не должно доходить до человеческого ревью (T0.2, T0.3, T1.3).
- **`autoreview` skill с few-shot примерами** — командный skill с конкретными парами анти-кода и реф-кода из истории команды, агент прогоняет его на собственном diff до пуша (T2.2).
- **Команда агентов-ревьюеров** — несколько специализированных reviewer-агентов (стиль, безопасность, архитектура) проходят по diff в параллель, человек видит структурированный summary (T3.1 + T3.2; production-пример — Cloudflare AI code review).

Открытый вопрос: на каком объёме команды и зрелости репо fast-review оправдан как командное правило, а не индивидуальный эксперимент.

#### Тесты: добавляемость и изменяемость

Темы, требующие отдельного обсуждения с тестировщиками:

- **Моки внешних зависимостей с кэшированием** — один раз обратились к реальному API, закэшировали ответ, далее гоняем тесты на кэше. Снижает стоимость локального прогона и убирает flakiness, но требует протокола invalidation (когда схема внешнего API изменилась — кэш сбрасывается).
- **Как агент запускает e2e** — есть ли изолированное окружение, сколько занимает полный проход, что считается smoke-набором для агента. Без этого e2e фактически выпадают из feedback loop агента (T0.2 не закрывается).
- **Разделение зоны ответственности с QA** — что тестировщики делают руками, что автоматизируется; какие тесты пишет агент при добавлении фичи, какие остаются на человеке.

#### Разделяемость на команды

Рамка: агенты не знают о командах напрямую — они работают на уровне пакетов и модулей. Командное разделение проявляется через структуру кода и локальные артефакты, а не через "знание про команду".

Что это значит на практике:

- **Границы пакетов и модулей**, проверяемые механически (T1.3) — команда владеет своим поддоменом, агент при работе в нём автоматически подгружает релевантный контекст.
- **Локальный `CLAUDE.md` / `AGENTS.md` команды** в директории её модулей — специфичные конвенции, anti-patterns, ссылки на эталонные файлы. Уровень overrides уже описан в T2.1 и в таблице overrides ниже; здесь фиксируем, что это **основной носитель командной специфики** для агента.
- **Командные skills** для упрощённого доступа к функционалу модуля — типичные операции, helpers, паттерны интеграции (T2.2, module-specific skills; см. также Случай Б1 про gap в знании репо).

Открытый вопрос — governance: кто owner локального AGENTS.md и skills команды, как разрешаются конфликты с глобальным слоем, как cross-team skills попадают из команды в общий репо.

## Concerns команды

> Типичные (реальные командные) боли работы с coding-agents.

### Случай А: агент забывает важные инструкции

**Проблема**: разработчик ставит в промпте правило ("новый компонент только если нет существующего, сначала проверь что нет дублей"), агент его держит первые сообщения, потом игнорирует — особенно после компакции или на длинной сессии.

**Что делаем:**

1. **Вносим правило в `CLAUDE.md` / `AGENTS.md`**, причём не как инструкцию, а как обязательный workflow-шаг: "перед созданием нового компонента — обязательный поиск по семантическим ключевым словам, результаты приложить в plan-секции ответа". Если ожидаемый шаг отсутствует в выводе — откат. Это T1.4 + правильная формулировка.
2. **Держим источник правды для фичи** — `system-design.md` или `docs/features/<feature>.md` — куда вносим что строим, какие компоненты переиспользуем (если понятно на этапе проектирования), какие критерии приёмки, что out-of-scope. Агент при работе над фичей читает спеку первой, а не выводит реквайерменты из своей головы. Это T1.7.
3. **Сложные фичи всегда через `/plan`**, никогда напрямую через "сделай мне X". План показывается человеку до старта работы, человек проверяет (хотя бы по диагонали), что план соответствует его пожеланию — например, в плане нет дублирования существующего; далее агент кодит уже по согласованному плану.

Замечание: правила про "не дублировать существующее" — довольно трудные для агента в принципе, потому что для их выполнения он должен сначала доказать отсутствие старого. Поэтому формулировка "запусти grep, приложи результаты" работает потенциально заметно лучше, чем "переиспользуй существующее" — наблюдаемый шаг. Если поведение не закрывается даже после T1.4 + T1.7 — следующий уровень это mechanical sensor (кастомный pre-commit hook или ESLint rule, который ловит создание файла с похожим API на уже существующий) — это конкретная имплементация T1.3.

**Какие пункты плана это закрывают:** T1.4, T1.7 — основная линия защиты. T1.3 — fallback, если поведенческие меры недостаточны.

### Случай Б1: агент не подтянул существующий helper

**Проблема**: модуль предоставляет helper, но агент его не находит и пишет inline-замену. Это не оверинжиниринг и не нарушение архитектурных границ — это **gap в знании репозитория**. Агент **не знал** изначально, в отличие от Случая А, где агент **знал и забыл**.

**Конкретный пример из практики.** Агент должен был обработать ошибку "графа не существует". Вместо использования существующего `graphErrors`-helper из `~/modules/graph/core/graph-errors`, агент inline определил собственный type и захардкодил строки:

```typescript
// Что сделал агент (плохо):
type GraphNotExistError = {
  error: 'graph.not-exist'
  ns: 'graph'
}
// ...
return resultErr({
  error: 'graph.not-exist',
  ns: 'graph',
})
```

```typescript
// Как должно быть (использовать существующий helper):
import { graphErrors } from '~/modules/graph/core/graph-errors'
// ...
return resultErr(graphErrors({ code: 'not-exist' }))
```

**Что делаем:**

1. **Module-specific skill** (`graph-module.skill.md` с публичным API модуля), который подгружается, когда агент работает в директории модуля. В нашем кейсе skill бы явно указал на `graphErrors` helper. Это T2.2.
2. **Документ-индекс** с указателем "графовые helpers — в `modules/graph/core/`" — превентивно работает при start-of-session loading. Это T1.6.
3. **Если ошибка повторяется** — добавляем явную запись про этот конкретный helper в Module README или локальный AGENTS.md ("для ошибок этого модуля — `graphErrors`, не определяй inline-типы"). Это уже **anti-pattern log** для конкретного случая (T1.4 в module-scope), а не стартовая мера — потому что добавление каждого helper-а в README не масштабируется, делаем только для тех, на которых агент уже спотыкался.

**Какие пункты плана это закрывают:** T1.6, T2.2 — основная линия защиты. T1.4 (в module-scope) — fallback при повторных багах.

### Случай Б2: оверинжиниринг

**Проблема**: агент пишет код с лишними слоями абстракции, ненужными интерфейсами, фабриками, преждевременной generalization. Не потому что не знает API, а потому что **по умолчанию пишет "правильный" код по учебникам** — модель обучена на корпусе enterprise-кода, и без явных контр-сигналов пишет, как будто это будущий монолит на 100k LOC.

**Что делаем:**

1. **Few-shot анти-примеры в `AGENTS.md`** — желательно локально в модулях. Не "пиши простой код", а конкретные пары анти-кода и реф-кода с указанием на reference-файлы в репо. Это первый шаг — самый дешёвый и сразу даёт эффект. Это T1.1 + T2.1.
2. **Reference-файлы как эталон стиля** — "стиль кода в этом проекте — см. `src/foo/bar.ts` как эталон". Агенты заметно лучше копируют существующий стиль, чем следуют абстрактным правилам.
3. **Style/taste skills** для того, что не ловится правилами линтера — командные в репо, личные в `~/.claude/skills/`. См. T2.2.
4. **Прямой запрет инициативы** — в **личном** `CLAUDE.md`/`~/.claude/CLAUDE.md`: "не делай рефакторинг, если не просили; не выноси в отдельный файл функцию, используемую один раз; не делай интерфейс, если у него один implementation". В личном, потому что это вопрос вкуса — у разных людей в команде разные предпочтения по тому, насколько агент должен проявлять инициативу.
5. **Опционально — кастомный code-review skill** с few-shot примерами оверинжиниринга из истории команды. Перед коммитом агент обязан прогнать его на собственном diff. Это T2.2 + T3.1.

Замечание про вкусовщину: вкусовщина — самое сложное для регулирования механическими правилами. Style/taste skills — естественное место для накопления "корпоративного вкуса", который сложно сформулировать в виде линтер-правил.

**Какие пункты плана это закрывают:** T1.1, T2.1, T2.2 — основная линия. T3.1 — fallback.

### Случай Б3: ревью становится невалидным

**Проблема**: PR содержит так много изменений, что ревьюер физически не может прочитать всё внимательно, проверки становятся формальными, баги проскакивают. По данным [Anthropic Code Review study](https://medium.com/data-science-in-your-pocket/code-review-for-claude-code-claude-code-can-now-review-prs-b281253cc61c), PR > 1000 строк содержат issues в 84% случаев со средним 7.5 находок; PR < 50 строк — в 31% случаев со средним 0.5 находок. Чем меньше PR, тем меньше дефектов и тем выше качество ревью.

**Рамки**: эта практика пока пробуется как **эксперимент в личных `CLAUDE.md`** конкретных разработчиков. Если работает — переезжает в командные правила и hooks. Не вводим сразу как командное правило, потому что неочевидно, какие именно лимиты подходят нашей кодовой базе.

**Что пробуем:**

1. **"Один PR — одно изменение"** в личном CLAUDE.md: не "feature + рефакторинг + чистка имён + новые тесты" в одном PR. Структурный summary в PR description (что изменилось, что НЕ изменилось).
2. **Diff size limits** — ориентир ~400 строк (по индустриальным данным, оптимум 200–400). Сначала как мягкое правило в личном CLAUDE.md, потом — как hook, блокирующий PR > N строк.
3. **Опционально — pre-review агент** (T3.1, Generator-Evaluator). Второй агент с другим system prompt проходится по diff с критериями оверинжиниринга / нарушения конвенций до того, как ревьюер-человек его увидит. Production-пример имплементации — [Cloudflare Engineering Blog](https://blog.cloudflare.com/ai-code-review/) (multi-agent code review поверх OpenCode).

**Какие пункты плана это закрывают:** T0.1 (правила в личном CLAUDE.md), T0.3 (hooks), T3.1 (pre-review агент).

## Командные и индивидуальные overrides

AI-native харнес — не обязательно монолитный. Рабочая модель — **многослойная конфигурация**, где у каждого слоя своя зона ответственности и своя политика изменения:

| Уровень | Где живёт | Кто меняет | Зачем |
|---|---|---|---|
| **Глобальный (project-wide)** | `AGENTS.md`/`CLAUDE.md` в корне, `docs/index.md`, mechanical rules, общие skills | RnD LLM lead / архитектор | Архитектурные инварианты, общекомандные правила, must-have anti-patterns |
| **Кросс-команды (cross-team shared)** | Cross-cutting skills (data extraction, vendor integration, общая infra) в репо | RnD LLM lead / архитектор; либо owner домена через PR с ревью RnD LLM lead | Знания, нужные нескольким командам |
| **Поддоменный (subteam/module)** | `AGENTS.md` в `src/payments/` и т.п., module-specific skills, module README | Owner поддомена | Специфика модуля, локальные конвенции, свои anti-patterns |
| **Личный (per-developer)** | `~/.claude/CLAUDE.md`, `~/.claude/skills/` (не коммитятся в репо) | Сам разработчик | Персональные предпочтения, рабочие гипотезы, временные experimental tweaks |

Примеры легитимных индивидуальных overrides — operational, не вкусовые:

- "Всегда вызывай `/code-review` skill после завершения задачи перед PR."
- "При работе с миграциями — сначала покажи план, потом генерь."
- "Не запускай `git push --force` без явного подтверждения, даже если я попрошу."
- "Перед коммитом запусти `pnpm verify` и приложи вывод."
- "Не правь файлы в `migrations/` без подтверждения."

Что важно держать в голове:

1. **Глобальный слой — единственный, который реально enforced.** Только то, что в общем `AGENTS.md`/`CLAUDE.md` плюс mechanical rules — гарантировано работает у всех.
2. **Личные overrides не должны переопределять архитектурные инварианты.** Если разработчик отключил у себя правило "не импортировать infrastructure из domain" — это ловится на CI (mechanical rule), а не на доверии.
3. **Поддоменные overrides могут быть строже глобальных, но не мягче.** В `src/payments/` можно добавить "обязательно audit-log на каждое изменение баланса", но нельзя снять глобальное правило.
4. **Личные dotfiles стоит шарить в команде как inspiration.** Хорошие персональные паттерны быстро проявляют себя как кандидаты в общий AGENTS.md или кросс-командные skills.

⚠ **Антипаттерн**: dichotomy "у нас общий AGENTS.md, и больше ничего не нужно". Реальные команды быстро упираются в то, что у разных подкоманд разные циклы, разный риск-профиль, разные привычки. Без явного механизма локальных overrides люди начинают **скрытно** переписывать общие правила в личных промптах — и общая дисциплина деградирует.

## Other: метрики, антипаттерны, связь с дисциплинами

### Метрики, которые имеет смысл отслеживать

Чтобы понимать, работает ли вложение в AI-native:

- **Доля PR, мерджаемых без правок.** Растёт — харнес работает.
- **Среднее количество итераций агента до зелёного CI.** Падает — feedback loop эффективен.
- **Количество правил в AGENTS.md и частота их добавления.** Замедление роста — большинство anti-patterns закрыты.
- **Время от feature spec до открытого PR.** Главный business outcome.
- **Регрессии static-analysis warnings и cognitive complexity.** Без guardrails агенты заметно увеличивают эти метрики (порядка +18% по warnings и +39% по complexity) — твой харнес должен это удерживать.
- **% знаний в репо vs знаний в чатах / головах.** Качественно — но важно для долгосрочной agent legibility.

### Антипаттерны at scale

- **AGENTS.md превращается в портянку >2000 строк.** Агент не справляется. Решение — иерархия, skills с progressive disclosure.
- **Slack-обсуждения архитектуры, не доходящие до `decisions/`.** Знание умирает для агента. Дисциплина: каждое обсуждение архитектуры заканчивается ADR.
- **Промпт-инструкции вместо CI gates для критичных правил.** "Просьба" не остановит агента. Только mechanical rule.
- **Sub-agents с полным parent context.** Ломает context firewall.
- **Скаффолдинг под устаревшую модель.** Раз в 1–2 релиза модели — ревизия харнеса. Anthropic регулярно удаляют целые слои harness'а после апгрейда модели.
- **Бесконтрольный `npm install` skills.** Уже зафиксированы случаи malicious skills в публичных регистрах.

### Связь с другими дисциплинами

| Дисциплина | Соотношение |
|---|---|
| **Harness engineering / Agentic harness** | Две стороны одного сдвига: AI-native repo описывает мир, в котором живёт агент; harness engineering — runtime агента (инструменты, hooks, sub-agents, контекстные политики) |
| **Clean Architecture / DDD** | AI-native использует те же принципы (однонаправленные зависимости, separation of concerns), но требует механической проверки правил, а не "договорённости в команде" |
| **DevOps / Platform Engineering** | AI-native добавляет агента как первого класса пользователя platform-а (наряду с человеком) |
| **Documentation as Code** | AI-native расширяет до executable spec и runtime context, который читают агенты |
| **Code Review** | AI-native: review всё больше делает агент-evaluator, человек — на стратегических решениях |

AI-native не отменяет ни одну из этих дисциплин — он их **усиливает и делает обязательными**, потому что ошибки масштабируются вместе с производительностью агента.
